## Summary

This phase involves removing the Header component from the BaseLayout and adjusting site styling to accommodate the structural change. The research confirms this is a straightforward component removal task in Astro with predictable CSS adjustments.

The standard approach is to remove the component import and usage from BaseLayout.astro, delete the Header component file, and adjust CSS spacing/layout to compensate for the removed fixed header. Since the current implementation uses a fixed header with margin-top compensation on the main element, removing it requires adjusting the main element's top margin.

**Primary recommendation:** Remove Header component from BaseLayout, delete the component file, adjust main element CSS to remove fixed-header compensation, run build verification to ensure no broken imports.

### Pattern 1: Component Removal from Layout
**What:** Remove component import and usage from Astro layout file
**When to use:** Simplifying site structure, removing navigation from single-page sites
**Example:**
```astro
---
// BEFORE
import Header from '../components/nav/Header.astro';
import Footer from '../components/common/Footer.astro';
---
<body>
  <Header />
  <main><slot /></main>
  <Footer />
</body>

// AFTER
import Footer from '../components/common/Footer.astro';
---
<body>
  <main><slot /></main>
  <Footer />
</body>
```

**Source:** [Astro Layouts Documentation](https://docs.astro.build/en/basics/layouts/) - Context7 verified

### Pattern 2: Fixed Header Compensation Removal
**What:** Adjust main element spacing when removing fixed-position header
**When to use:** After removing a position: fixed header element
**Example:**
```css
/* BEFORE - Compensating for fixed header */
main {
  margin-top: calc(var(--space-lg) + var(--space-md) + var(--border-width));
  padding: var(--space-lg) var(--space-md);
}

/* AFTER - No fixed header to compensate for */
main {
  padding: var(--space-lg) var(--space-md);
}
```

**Source:** [Stop Using Fixed Headers and Start Using Sticky Ones](https://dev.to/luisaugusto/stop-using-fixed-headers-and-start-using-sticky-ones-1k30)

### Pattern 3: Neo-Brutalist Single-Page Layout
**What:** Simple, bold layout with visible structure and minimal navigation
**When to use:** Single-page landing sites with neo-brutalist aesthetic
**Design principles:**
- Keep spacing visible and intentional
- Use bold, balanced negative space
- Show the layout grid (borders, clear containers)
- Minimize navigation chrome to focus on content

**Source:** [Neobrutalism: Definition and Best Practices](https://www.nngroup.com/articles/neobrutalism/)

### Anti-Patterns to Avoid
- **Leaving unused imports:** Always remove component imports when deleting component usage - Astro build will warn but won't fail
- **Forgetting CSS cleanup:** Fixed headers create CSS dependencies (margin-top, padding-top) that must be removed
- **Partial deletion:** Delete both the component file AND all references/imports to prevent confusion

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Checking for unused imports | Manual grep | `astro check` | Type-checks and validates Astro files |
| Build verification | Manual testing | `npm run build` | Catches import errors, missing files |
| CSS layout testing | Manual resize | Browser DevTools responsive mode | Reliable viewport testing |

**Key insight:** Astro's build process catches most structural errors (missing imports, deleted components still referenced). Always run a full build after component removal to surface any missed references.

## Common Pitfalls

### Pitfall 1: Orphaned CSS Selectors
**What goes wrong:** CSS targeting the removed header (e.g., `.site-header`) remains in files
**Why it happens:** Global or component-scoped styles may reference the header
**How to avoid:** Search entire codebase for header-related class names before committing
**Warning signs:** Unused CSS in global.css or tokens.css, no visual changes after "update"

### Pitfall 2: Body/HTML Flexbox Layout Breaking
**What goes wrong:** Body uses flexbox to push footer to bottom; removing header can affect layout
**Why it happens:** Three-part flex layout (header, main flex:1, footer) becomes two-part
**How to avoid:** Verify footer still sticks to bottom after header removal, test on short content pages
**Warning signs:** Footer floating mid-page, excessive whitespace above footer

### Pitfall 3: Incorrect Margin Compensation Removal
**What goes wrong:** Removing too much or too little top margin from main element
**Why it happens:** Fixed header compensation in BaseLayout.astro line 48: `margin-top: calc(var(--space-lg) + var(--space-md) + var(--border-width))`
**How to avoid:** Calculate exact compensation being removed (in this case: 2rem + 1.5rem + 3px = 3.5rem + 3px)
**Warning signs:** Content touching viewport top edge, or excessive whitespace at page top

### Pitfall 4: Forgetting Responsive Breakpoints
**What goes wrong:** Header has mobile/desktop variants; CSS may have orphaned media queries
**Why it happens:** Header component (lines 138-163) has `@media (min-width: 768px)` block
**How to avoid:** Check for media queries targeting header classes in all CSS files
**Warning signs:** Build succeeds but unnecessary CSS remains, DevTools shows unused selectors
