# Phase 1.1: Update Page Structure - Research

**Researched:** 2026-01-29
**Domain:** Astro file-based routing, site structure refactoring
**Confidence:** HIGH

## Summary

This phase involves converting a multi-page Astro site to a single-page site by removing pages from the `src/pages/` directory and updating navigation references. The work is straightforward in Astro due to its file-based routing system where each file in `src/pages/` automatically becomes a route.

The standard approach is to:
1. Delete unwanted page files from `src/pages/`
2. Update all navigation components to remove links to deleted pages
3. Search and remove any other references to deleted pages (imports, href attributes, etc.)
4. Verify the build completes successfully without errors
5. Optionally verify no broken links exist using link validation tools

**Primary recommendation:** Delete page files, grep the codebase for references to removed routes, update navigation components, and run `npm run build` to verify no broken links or references remain.

## Architecture Patterns

### Recommended Project Structure
```
src/
├── pages/           # File-based routing - each file becomes a route
│   └── index.astro  # Root route (/)
├── components/      # Reusable components
│   └── nav/         # Navigation components that link between pages
└── layouts/         # Page layouts
```

### Pattern 1: File-Based Routing in Astro
**What:** Astro automatically creates routes based on file structure in `src/pages/`
**When to use:** All Astro projects - it's the default routing system
**Example:**
```diff
# File-to-route mapping
src/pages/index.astro        -> mysite.com/
src/pages/about.astro        -> mysite.com/about
src/pages/products.astro     -> mysite.com/products

# After deletion:
src/pages/index.astro        -> mysite.com/
(about.astro deleted)        -> 404
(products.astro deleted)     -> 404
```
Source: [Astro Routing Docs](https://docs.astro.build/en/guides/routing/)

### Pattern 2: Standard HTML Navigation
**What:** Astro uses standard `<a>` elements for navigation, no special framework components
**When to use:** Always - Astro doesn't provide a custom Link component
**Example:**
```astro
<!-- Source: https://github.com/withastro/docs -->
<nav>
  <a href="/">Home</a>
  <a href="/about">About</a>  <!-- Delete this when removing about page -->
</nav>
```

### Pattern 3: Systematic Reference Cleanup
**What:** Search for all references to deleted routes and remove/update them
**When to use:** After deleting any page files
**Example:**
```bash
# Find all references to deleted pages
grep -r "/about\|/products\|/music" src/

# Common places to check:
# - Navigation components (Header, Footer)
# - Page content (href attributes, imports)
# - Layouts (any hardcoded links)
```

### Anti-Patterns to Avoid
- **Leaving broken links in navigation:** Always update nav components when removing pages
- **Not verifying the build:** Deleted pages might be imported elsewhere, causing build failures
- **Manual link checks only:** Use `grep` to systematically find all references

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Broken link detection | Custom link crawler | astro-link-validator or grep | Edge cases (external links, assets, srcset), performance optimization |
| 404 redirect management | Custom redirect logic | Astro's built-in 404.astro | Framework handles this automatically |
| File search | Manual file browsing | grep/ripgrep via Bash | Faster, more reliable, finds all references |

**Key insight:** Astro's file-based routing makes page removal simple - just delete the file. The complexity is in finding and updating all references, which is best done with search tools.

## Common Pitfalls

### Pitfall 1: Broken Internal Links After Page Removal
**What goes wrong:** Pages are deleted but navigation components still link to them, creating 404s
**Why it happens:** Navigation is typically in shared components (Header.astro) that aren't automatically updated when pages are deleted
**How to avoid:**
1. Search the entire codebase for references: `grep -r "/deleted-route" src/`
2. Update or remove all navigation links to deleted pages
3. Build the site to catch import errors: `npm run build`
**Warning signs:** Build succeeds but deployed site has broken navigation links

### Pitfall 2: Orphaned Component Imports
**What goes wrong:** Deleted pages are still imported in other files, causing build failures
**Why it happens:** Components, layouts, or other pages may import deleted page components
**How to avoid:**
1. Search for component name: `grep -r "DeletedPage" src/`
2. Remove or update imports before deleting the file
3. Verify with `npm run build`
**Warning signs:** TypeScript/build errors about missing modules

### Pitfall 3: Disrupted Site Architecture Without Proper Verification
**What goes wrong:** Internal links, breadcrumbs, or navigation structure breaks after removal
**Why it happens:** Pages may be referenced in multiple places (nav, footer, breadcrumbs, content)
**How to avoid:**
1. Document all places a page is referenced before deletion
2. Use systematic search: `grep -r "page-name" src/`
3. Update all references simultaneously
4. Verify build succeeds
**Warning signs:** 404 errors, broken navigation flows, poor user experience

Source: [11 Things to Consider When Removing Content From Websites](https://www.cyber-duck.co.uk/insights/removing-old-content-from-your-website-11-things-to-consider)

### Pitfall 4: Not Testing the Build
**What goes wrong:** Changes work in dev mode but fail in production build
**Why it happens:** Astro's dev server is more forgiving than the build process
**How to avoid:** Always run `npm run build` before committing page deletions
**Warning signs:** Deploy failures, broken production site

## Code Examples

Verified patterns from official sources:

### Removing a Page and Updating Navigation
```astro
<!-- Before: src/components/nav/Header.astro -->
<nav>
  <a href="/">Home</a>

  <a href="/products">Products</a>
```

### Searching for References Before Deletion
```bash
# Find all references to pages about to be deleted
grep -r "/about\|/products\|/music" src/

# Check for component imports
grep -r "from.*about.astro\|from.*products.astro" src/

# Verify no broken links after changes
npm run build
```

```bash

# Build command that validates entire site
npm run build

# Output location
ls dist/  # Should contain only index.html for single-page site
```

## Open Questions

1. **Should we implement automated link validation?**
   - What we know: Tools like astro-link-validator exist and integrate with builds
   - What's unclear: Whether the overhead is worth it for a simple single-page site
   - Recommendation: Skip for now (manual verification sufficient); add if site grows multi-page again

2. **Should we create a custom 404 page?**
   - What we know: Astro supports 404.astro in src/pages/
   - What's unclear: Whether it's needed for a single-page site
   - Recommendation: Not needed for single-page site; consider if adding pages later
